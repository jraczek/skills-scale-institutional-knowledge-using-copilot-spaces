# OctoAcme - Role Handoff Checklist

## Purpose
Reduce ambiguity at role boundaries by defining what must be exchanged between roles before work can move forward.

## When to use
- During planning before execution starts.
- At sprint boundaries for high-risk or cross-team work.
- Before release go/no-go decisions.

## Planning Handoff Checklist

### Product Manager -> Engineering Lead and Developers
- [ ] Problem statement and expected user outcome are documented.
- [ ] Acceptance criteria are testable and unambiguous.
- [ ] Priority and scope boundaries are explicit.
- [ ] Open assumptions and unresolved decisions are tracked.

### Product Manager and UX Research / Design -> Delivery Team
- [ ] Key user flows, constraints, and edge cases are shared.
- [ ] Usability risks are documented with mitigation options.
- [ ] Success signals for post-release validation are defined.

### Engineering Lead -> Project Manager
- [ ] Technical dependencies and sequencing risks are documented.
- [ ] Architecture decisions and trade-offs are recorded.
- [ ] Delivery risks that may affect milestones are escalated.

### Engineering Lead and Developers -> QA / Testing
- [ ] Test strategy and regression scope are aligned.
- [ ] Feature flags, test data, and environments are ready.
- [ ] Non-functional checks needed for release are identified.

### Security / Compliance -> Project Manager and Engineering Lead
- [ ] Required controls, evidence, and approvals are listed.
- [ ] Compliance checkpoint timing is aligned to milestones.
- [ ] High-risk findings have owners and due dates.

## Release Handoff Checklist

### Delivery Team -> Customer Support / Operations
- [ ] Release notes include customer impact and known limitations.
- [ ] Runbooks and escalation contacts are updated.
- [ ] Monitoring and alert expectations are shared.

### Project Manager -> Executive Sponsor / Stakeholders
- [ ] Release readiness summary includes risks and mitigations.
- [ ] Decision needed (inform, approve, or escalate) is explicit.
- [ ] Timeline changes and impact are clearly communicated.

## Exit Criteria
- [ ] All required handoffs completed and documented.
- [ ] Any exceptions have a named owner and target date.
- [ ] Open risks are captured in the risk register.
